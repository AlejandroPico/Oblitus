Este artículo procede de un documento fuente situado en articulos/. En una publicación real, el contenido puede venir de Word, Markdown, HTML o PDF y se convierte automáticamente durante el despliegue.

1. Propósito

Oblitus est scientia nace como un archivo editorial independiente dedicado a recuperar, ordenar y redivulgar conocimiento científico y técnico. La intención no es competir con una noticia breve, sino construir piezas de largo recorrido: investigaciones, ensayos, series documentales y materiales de consulta.

La web busca combinar el tono de una revista seria con la libertad técnica de una página propia: texto extenso, audio, imágenes, esquemas, tablas, visualizaciones y pequeñas interfaces interactivas cuando el contenido lo requiera.

2. Línea editorial

La línea editorial prioriza la claridad, la profundidad y la trazabilidad de las ideas. La ciencia no se entiende aquí solo como una colección de datos, sino como una forma de mirar el mundo, contrastar afirmaciones y conectar disciplinas.

  • Explicaciones largas cuando el tema lo necesite.
  • Contexto histórico y conceptual antes de la conclusión rápida.
  • Separación clara entre hechos, interpretación y opinión editorial.
  • Uso de recursos multimedia cuando aporten comprensión real.

3. Formato de publicación

La primera versión de la web no necesita base de datos. Los artículos se guardan como documentos en la carpeta articulos/ y un proceso automático genera el índice que lee la portada. Así se obtiene una experiencia similar a un blog, pero con coste cero y control completo sobre el diseño.

Documentos largos

Para textos densos, el flujo recomendado es escribir en Word o LibreOffice usando estilos semánticos: Título 1, Título 2, listas, tablas y pies de imagen. Después se sube el archivo .docx al repositorio y se deja que la acción de GitHub genere la versión web.

Artículos interactivos

Cuando un artículo necesite animaciones, controles o programas JavaScript internos, lo más limpio será crearlo como una carpeta propia dentro de articulos/, con su index.html y sus recursos asociados.

4. Futuro del proyecto

Esta base permite empezar sin servidor. Más adelante, si el proyecto crece, se podrá migrar a un CMS real, a un sistema con base de datos o a un generador estático más avanzado sin perder la arquitectura editorial ya creada.
